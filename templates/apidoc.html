eGSIM is a web service for selecting and testing Ground Shaking Intensity Models (GSIM) in Europe on top of gmpe-smtk, a Python and OpenQuake-based toolkit for analysis of strong Motions and interpretation of Ground Motion Perdiciton Equations (GMPE). As server-side web application programming interface (API), the goal of eGSIM is to provide most of the toolkit functionalities via several publicly exposed endpoints (URLs) to a defined and easy to use requestâ€“response message system.

General features

eGSIM is implemented along the lines of typical web services (see e.g. the FDSN protocol): requests can be issued by a client (eGSIM user) in several ways such as e.g. command line applications like wget or inside client code with dedicated libraries (e.g. Python's urllib.request). Requests require an endpoint, i.e. an URL denoting a specific service, a method (GET vs POST) and usually parameters for tuning the functionality. The web server processes the request and returns a response to the client in specific formats described in the remainder of this document.

Requests endpoints

The following base URI pattern is to be used for each request:

{{ baseurl }}/<service>

where <service> is one of the following:

  • {{ gsimsel }} for Gsim selection
  • {{ trellis }} for generating Trellis plots data
  • {{ residuals }} for Residual analysis
  • {{ test }} for Testing

Requests methods

eGSIM services will be invoked using a subset of REST and HTTP methods. In particular all services will be invoked using the HTTP GET method whenever possible. As for FDSN web services, the HTTP POST method might be invoked for the cases where the selection parameters are potentially large in size (typically, POST request are used to send data to create/update a resource on the server. Unless otherwise stated, this is generally not the case here).

GET method

Requests issued with the GET method simply concatenate the parameters names and values directly in the URL:

{{ baseurl }}/<service>?name1=value1&name2=value2

POST method

Requests issued with the POST should use the usual URI pattern: {{ baseurl }}/<service> and provide the parameters set by means of additional data as text in JSON or YAML format. The way the POST request has to be built with its data is responsibility of the user as it depends on the client software. For instance, this code snippet shows how to build a request using Python (see HOWTO Fetch Internet Resources Using The urllib Package for details): 

import json
import yaml  # needs PyYAML installed

# imports for Python3  (comment/remove in Python2):
from urllib.request import Request, urlopen
from urllib.error import URLError, HTTPError

# imports for Python2 (comment/remove in Python3):
from urllib2 import Request, urlopen, URLError, HTTPError

# 1) Request using the GET method:
req = Request("write here your eGSIM endpoint URL with encoded parameters")

# 2a) Request using the POST method, assuming JSON text file:
with open('/path/to/my/file.json', 'r') as stream:
    jsondata = json.load(f)
req = Request("write here your eGSIM endpoint URL", data=jsondata)

# 2b) Request using the POST method, assuming YAML or JSON text file
#     (YAML is a superset of JSON and thus this works also with JSON files):
with open('/path/to/my/file.yaml', 'r') as stream:
    yamldata = yaml.safe_load(stream))
req = Request("write here your eGSIM endpoint URL", data=yamldata)

Request parameters

Request parameters can be input in three ways: in the URL of a GET request, in a YAML- or JSON-formatted text file in case of POST request, or via input components of the web Graphical User Interface (GUI) thorugh a web browser. eGSIM supports the following parameter types:

  • numeric
  • string
  • boolean
  • UTC date-time (in ISO format without timezone, e.g.: 2016-01-30T23:15:11 or 2016-01-31)
  • arrays (vectors) of the previous types
  • numeric ranges (i.e., equally spaced numeric arrays) in the Matlab(c) format start:step:end (e.g. 0:0.1:0.3 results in the sequence of values 0, 0.1, 0.2, 0.3)

For POST requests, most types are supported in YAML and JSON except numeric ranges, which must be input as strings with quotes (e.g. "0:0.1:0.3"). JSON does not support also date-times, which thus need to be input in the same way, e.g.: "2016-01-31T23:15:00" (in YAML, date-times as quoted strings are permitted but not mandatory).

For GET request URLs, and from within an input component of the Web GUI, spaces, commas (,) and colons (:) might be treated as special characters: as already described, the colon acts as separator for numeric ranges, whereas spaces and commas act as element separators in arrays (but only the comma is valid in GET requests URLs, being spaces not allowed by definition). Arrays can be input with or without leading and trailing square brackets, and no parameter value needs to be quoted.

For instance, assuming a general example service requiring three numeric parameters (dip, scalar, Vs30, array and ranges of magnitudes) the user input, depending on the request's method, should look like:

Response data

The response data type of a successfully fullfilled request might depend on the service used and the parameter issued in the request, but all eGSIM services can return data at the least in CSV and JSON formatted text (bytes sequences): for client code processing response data, both formats are widely supported in most programming languages. For JSON, example are the Matlab jsonencode function or the Python builtin json library. Csv format has the advantage to be easily visualizable in widely used spreadsheet softwares (like e.g. LibreOffice). JSON is usually preferable in web applications: the eGSIM web GUI for instances requests data in this format.

Unsuccessful requests might fail for several reasons. In case of client or server errors, such as e.g. bad or missing parameters, eGSIM always returns responses in JSON formatted byte strings following the google API specification. Example: 

{
	"code": 400,  # the HTTP error numeric code ≥400
	"message": "input validation error",  # the HTTP error message
	"errors": [  # optional array of parameter specific errors:
		{
	    	"domain": "magnitude",  # the parameter name
	    	"message": "This field is required.",  # the parameter-specific
	    										   # error message
	    	"reason": "required"  # the parameter-specific
	    						  # error reason
	  	},
	  ...
	]
}

For instance, this code snippet modified from the standard Python how-to reference shows how to issue a request to a eGSIM service and catch potential errors in both Python2 and 3 (tested with Python 2.7 and 3.6): 

import json

# imports for Python3  (comment/remove in Python2):
from urllib.request import Request, urlopen
from urllib.error import URLError, HTTPError

# imports for Python2 (comment/remove in Python3):
from urllib2 import Request, urlopen, URLError, HTTPError

req = Request("write here your eGSIM endpoint URL")
try:
    data = json.loads(urlopen(req))
except HTTPError as e2:
    # handle HTTP errors because of a server or client error, e.g.:
    print("Http error %d" % e2.code)
    # If you want to dig into details,
    # load the JSON formatted response into a Python dict:
    errordict = json.loads(e2.read())
    # now you can access fields such as
    # errordict.errors, errordict.message, ...
except URLError as e1:
    # handle general url errors like network errors. e.g.:
    print("Url error. Reason: %s" % e1.reason)

Trellis plots

Endpoint: {{ baseurl }}/{{ trellis }}

Trellis plots provide an understanding of the manner in which each GMPE characterises the ground motion scaling with respect to the properties of the seismic source, and the attenuation of the motion with respect to distance. By means of the underlying gmpe-smtk library, eGSIM can generate different plot types:

  • IMT vs Magnitude plots show how the ground motion intensity value scales with respect to the magnitude of the rupture.
  • IMT vs Distance plots allow to compare how the GMPEs describe the attenuation of strong motion with distance for a given magnitude.
  • Magnitude-Distance spectra let the user understand how a GMPE will scale the response spectrum with magnitude and distance.

In addition to viewing the scaling of the expected ground motion from the GMPE, it is also possible to view the scaling of the standard deviation, for a total of six choosable plot types.

Request

A request to the Trellis plot service requires a list of GMPEs and IMTs, the desired plot type and the parameters configuring the scenario (rupture), most of which are optional. As discussed, parameters can be given in the endpoint URL (GET method) or as separate text data in JSON or YAML format (POST method) and are:

Residual analysis

eGSIM, through the GMPE-SMTK library, offers the ability to compare observed ground motions with the GMPE model predictions using the GMPE implementations found inside OpenQuake. This is a particularly powerful tool as it permits, possibly for the first time, a hazard modeller to undertake the GMPE comparison using the exact same GMPE implementation found inside the seismic hazard software. There are several types of Residual plots:

Plot type x values (generated from the GMDB): Number of generated plots per IMT and per Gsim:

Residuals vs. MagnitudeScatter plot of magnitudes residuals vs see above

Request

A request to the Residuals plot service requires a list of GMPEs and IMTs, a Ground Motion database (GMDB) and the the desired plot type. As discussed, parameters can be given in the endpoint URL (GET method) or as separate text data in JSON or YAML format (POST method) and are:

{{ form_residuals | safe }}

Response

The number of generated Residuals plots will be three times the product of the requested Intensity measure types (parameter {{ param.imt }}) and the requested GMPEs (parameter {{ param.gsim }}, where three is the number of residual types per plot, which are "Intra Event", "Inter Event" and "Total". For decomposition of Intra-event residuals, the plots are still three but all labelled as 'IntraEvent' (what changes will be the y label and the data).
